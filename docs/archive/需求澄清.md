# LinkHub 需求澄清文档

## 1. 一句话定义

LinkHub 是一个跨平台可信设备 Mesh 互联系统。它让用户自己的 Android、iOS、iPadOS、Linux、macOS 和 Windows 设备互相信任、自动发现、自动连接，并在不同网络条件下组合局域网、蓝牙、NFC、有线网络、WebRTC、中继等通道，完成文本、文件、语音、视频和媒体播放协同。

最终愿景是：

> 用户的所有可信设备组成一个私有互联网络。A 可以直连 B，C 可以直连 A；当 C 不能直连 B 时，可以在授权和端到端加密保护下通过 A 中继，最终实现 C 到 B 的可靠通信。

第一版不要先追求“大而全的跨平台生态”，而是先证明一个核心体验：

> 两台已配对设备在同一局域网中可以自动发现彼此，并可靠地发送文本和文件。这是后续多设备 Mesh、音视频和媒体协同的地基。

## 2. 愿景能力地图

LinkHub 的长期能力可以分成六层：

1. 设备身份层：每台设备有稳定身份、可信关系和安全密钥。
2. 发现连接层：设备可以通过局域网、蓝牙、NFC、有线网络和公网信令发现彼此。
3. 路由传输层：系统能在直连、中继、多跳路径之间自动选择，并根据质量实时切换。
4. 数据协作层：支持文本、URL、文件、图片和离线队列。
5. 实时通信层：支持语音电话、视频通话和低延迟实时媒体。
6. 媒体协同层：支持跨设备无缝播放歌曲、播放控制迁移和设备间播放状态同步。

这些能力不应该一次性全部进入 MVP，而应该按阶段逐步实现。

## 3. 当前最应该解决的问题

用户在多设备之间传东西时，经常遇到这些麻烦：

- 不想每次都登录聊天软件、网盘或邮箱中转。
- 不想手动输入 IP、端口或临时搭服务。
- 不想因为 Wi-Fi 切换、设备睡眠、连接中断而重新开始。
- 希望设备只和自己信任的设备通信，而不是局域网里所有设备都可见可连。
- 希望多台设备可以像一个整体一样协作，而不是每个设备都是孤岛。

所以 LinkHub 的第一目标不是“远程控制”或“万能同步”，而是先建立一套可信连接和传输基础：

- 可信设备之间自动出现。
- 当前可达路径自动选择。
- 文件和文本能直接发送。
- 中断后尽量可恢复。
- 后续可以扩展到设备中继、语音视频和媒体播放协同。

## 4. 目标用户

### 第一阶段目标用户

- 同时使用 Windows、Linux、Android、iPad、macOS 等多台设备的人。
- 经常在自己设备之间传文件、图片、文本、链接的人。
- 愿意运行一个常驻工具来换取更顺畅设备互联体验的人。

### 暂时不是第一目标用户

- 只想远程控制另一台电脑的用户。
- 需要企业级设备管理、审计、权限分组的团队。
- 需要公网远程访问和云同步作为第一需求的用户。
- 期待完整 AirDrop 替代品且立刻覆盖所有平台细节的用户。

## 5. MVP 应该长什么样

MVP 的推荐形态是：

- 一个 Windows 桌面端作为第一个真实客户端。
- 一个 Rust Core 作为跨平台核心。
- 至少能在两台桌面设备之间完成配对、发现、状态展示、文本发送和文件发送。
- Linux 和 macOS 可以作为桌面端扩展目标。
- Android 先作为移动端第二阶段接入，不要在第一阶段同时处理太多平台差异。

第一版用户看到的核心界面可以很简单：

- 本机设备名和身份状态。
- 已信任设备列表。
- 每台设备的在线状态和当前连接方式。
- 发送文本入口。
- 发送文件入口。
- 最近传输记录和失败原因。

## 6. 第一版核心用户流程

### 流程 A：初始化本机

1. 用户首次打开 LinkHub。
2. 应用生成本机设备身份。
3. 用户设置或确认设备名称。
4. 应用开始监听局域网连接，并准备广播发现信息。

验收标准：

- 本机有稳定设备 ID。
- 重启应用后身份不变。
- 本机可以显示为“已准备好接收”。

### 流程 B：配对可信设备

1. 用户在设备 A 上发起配对。
2. 设备 A 显示二维码或短码。
3. 用户在设备 B 上确认同一个码。
4. 两台设备互相写入 Trust Store。

验收标准：

- 未配对设备不会自动进入可信列表。
- 配对后重启应用仍然记得对方。
- 配对确认码必须能防止误配陌生设备。

### 流程 C：自动发现

1. 两台已配对设备位于同一局域网。
2. 应用通过 mDNS/Bonjour 发现对方。
3. 发现结果只对已信任设备生效。
4. UI 显示设备在线和可连接地址。

验收标准：

- 同一局域网内发现时间目标不超过 3 秒。
- 陌生设备即使被扫描到，也不会显示为可操作可信设备。
- 设备离开网络后状态会过期。

### 流程 D：发送文本

1. 用户选择一台在线可信设备。
2. 输入一段文本或 URL。
3. 点击发送。
4. 对端收到并确认。

验收标准：

- 发送端能看到成功或失败。
- 每条消息有唯一 `message_id`。
- ACK 不匹配时不能误报成功。
- 重复消息不会被重复处理。

### 流程 E：发送文件

1. 用户选择一台在线可信设备。
2. 选择文件。
3. 应用按分片发送。
4. 对端写入临时文件。
5. 文件完整校验通过后再保存为最终文件。

验收标准：

- 文件名需要安全处理，不能写出接收目录。
- 文件大小和 SHA-256 必须校验。
- 传输中断时保留 `.part` 和 `.part.meta`。
- 再次发送同一个文件时可以从已确认 chunk 继续。

## 7. MVP 包含范围

第一版应该包含：

- 本机身份生成和持久化。
- Trust Store 可信设备保存/读取。
- 短码或二维码配对。
- mDNS 局域网发现。
- TCP 控制通道。
- 心跳和在线状态。
- 文本发送。
- 文件分片发送。
- 文件完整性校验。
- 基础断点续传。
- Windows 桌面端基础 UI。

## 8. 暂缓范围

第一版先不做：

- 公网云端中继。
- WebRTC NAT 穿透。
- 多跳 Mesh 路由。
- 语音电话。
- 视频通话。
- 无缝音乐播放。
- 跨设备播放控制。
- 远程控制。
- 投屏。
- 通知镜像。
- 插件系统。
- 多用户组织权限。
- 完整移动端后台体验。

这些不是不重要，而是会显著扩大项目复杂度。它们应该在局域网 MVP 稳定后再进入路线图。

## 9. 当前项目已经完成的部分

当前 Rust Core 原型已经覆盖：

- 设备身份模型。
- Ed25519 本机身份密钥骨架。
- 本机身份文件保存/读取。
- Windows DPAPI 安全身份文件保存/读取，支持通过 `secure:<path>` 参与配对和认证命令。
- 本机状态快照和 HTML 状态页生成，展示本机身份和可信设备列表。
- 可二维码化的 `linkhub-pair-v1` 配对邀请 payload。
- 基于配对 payload 的确认码计算和 Trust Store 写入 CLI，作为二维码/扫码流程的 core 等价物。
- Ed25519 握手挑战签名和验证骨架。
- 认证版 TCP 文本和文件发送原型：监听端通过 Trust Store 和签名验证发送端身份后才接收文本和文件。
- 心跳时间戳跟踪。
- 连接状态转换。
- 传输通道评分和路由选择。
- Trust Store 文件保存/读取。
- 配对短码确认骨架。
- mDNS 广播和扫描。
- 可信设备发现过滤。
- TCP HELLO 和 HEARTBEAT。
- 文本消息发送、ACK、重试和重复消息去重。
- 文件分片发送。
- `.part` 临时文件和 `.part.meta` 进度元数据。
- SHA-256 完整性校验。
- 基础断点续传。
- 本机端到端验收脚本，覆盖文本发送、完整文件发送、预置半成品断点续传、Windows DPAPI 安全身份文件、基于 payload 的配对确认、认证版文本发送、认证版文件发送和 HTML 状态页生成。
- Noise KK 双向认证和加密会话。
- Android JNI 桥接和真机前台监听服务。
- Android 与 Windows 在手机热点局域网内的双向加密文本互发。

这说明项目不是停留在概念阶段，已经有了可复用 core。下一步要从“命令行能力验证”转向“真实用户流程验证”。

## 10. 当前最关键缺口

按优先级排序：

1. Android 与 Windows 文件互发真机验收。
2. 可信设备地址自动发现和自动更新，避免手动输入 IP。
3. 二维码渲染 UI 和扫码接收流程界面，core 侧已有 payload、确认码和 Trust Store 写入命令。
4. 将认证能力从命令行原型接入真实桌面端用户流程。
5. Windows 桌面端基础 UI，替代 CLI listener 和 CLI sender。
6. Android 文件选择器、接收目录、通知权限和错误提示产品化。
7. macOS Keychain 和 Linux Secret Service 安全存储后端验证。
8. 防火墙、权限、多网卡、VPN 场景处理。

## 12. 长期阶段拆分

### 阶段 A：可信设备直连

- Windows / Linux / macOS 桌面端优先。
- 局域网自动发现。
- 可信配对。
- 文本发送。
- 文件发送。
- 断点续传。
- 状态展示。

### 阶段 B：移动端接入

- Android 接入。
- iOS / iPadOS 接入。
- 处理移动端后台限制。
- 手机和平板可以与电脑互传文本、文件和链接。

### 阶段 C：多路径连接

- Wi-Fi 和有线网络作为高速主链路。
- WebRTC 作为跨网络直连能力。
- 蓝牙用于附近发现、低速控制或兜底。
- NFC 用于快速配对和近场确认，不作为大文件主传输通道。
- 路由评分根据延迟、带宽、可靠性、电量和网络成本实时调整。

### 阶段 D：Mesh 和中继

- 每台可信设备可以在授权后作为中继节点。
- 支持 A 到 B、C 到 A、C 经 A 到 B 的通信模型。
- 支持多跳路由。
- 中继设备不能读取 payload 内容，必须保持端到端加密。

### 阶段 E：实时音视频

- 语音电话。
- 视频通话。
- 通话中设备切换。
- 优先考虑 WebRTC 或同等级实时媒体方案。

### 阶段 F：媒体播放协同

- 一台设备控制另一台设备播放。
- 播放队列同步。
- 音乐播放从手机切到电脑或平板。
- 播放状态在可信设备之间同步。

## 13. 设计约束和判断

- 支持越多连接协议越好是长期方向，不是第一版要求。
- 蓝牙和 NFC 更适合发现、配对、唤醒、低速控制，不适合作为大文件主传输。
- 语音和视频需要低延迟实时媒体栈，复杂度高，应在传输基础稳定后再做。
- Mesh 中继必须以授权和端到端加密为前提。
- iOS 和 iPadOS 后台能力有限，不能按桌面系统常驻代理来设计。
- 第一版优先做确定能闭环的局域网 TCP/QUIC 能力。

## 14. 需要你拍板的问题

这些问题决定产品方向，建议逐个确认：

1. 第一版是否只要求同一局域网可用？
2. 第一版是否必须有移动端，还是先把桌面到桌面跑通？
3. 文件发送是不是第一优先级，文本/URL 发送是否只是附带能力？
4. 是否允许用户手动确认每次接收文件，还是默认可信设备自动接收？
5. 接收文件默认保存在哪里？
6. 是否需要传输历史记录？
7. 是否需要设备分组，例如“我的设备”和“临时可信设备”？
8. 断点续传是否只支持同一个文件再次发送，还是要支持自动恢复未完成任务？
9. 安全性第一版要做到什么程度：短码确认即可，还是必须加密传输后才算完成？
10. Mesh 中继是否默认关闭，只对用户明确授权的设备开放？
11. 语音电话、视频通话和媒体播放协同哪个优先级更高？

## 15. 推荐产品判断

为了让项目尽快清晰，建议先采用这个判断：

> LinkHub 的长期方向是“跨平台可信设备 Mesh 互联系统”；第一版是“可信设备之间的局域网文件和文本直传工具”。

这个定义足够小，可以做完；也足够有价值，可以继续扩展。

第一版完成后，再向三个方向扩展：

- 移动端接入。
- 跨网络可达和 Mesh 中继。
- 语音视频和媒体播放协同。
