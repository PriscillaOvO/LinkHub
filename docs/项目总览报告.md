# LinkHub 项目总览报告

> 生成时间：2026-06-16（2026-06-16 更新：文件接收反馈与可复现 NDK 构建已落地，docs 已中文化重组）
> 适用对象：接手开发 / 准备重构的工程师

---

## 1. 项目一句话概述

LinkHub 是一个**跨平台可信设备互联原型**：让用户自己的 Android / iOS / Windows / macOS / Linux 设备先完成**安全配对**，再通过 **mDNS 局域网自动发现**，最终在**端到端加密通道（Noise KK + ChaCha20-Poly1305）**上可靠地互传文本和文件。长期愿景是发展成一个支持多跳中继、语音视频、媒体协同的私有设备 Mesh 网络，但当前阶段只聚焦于验证「身份 + 发现 + 加密传输」这条最小核心链路。

## 3. 技术栈分析

| 层 | 技术 | 说明 |
|----|------|------|
| 共享核心 | **Rust 2021** | 身份、加密、发现、传输全部在此实现，编译成 `lib`（给 Tauri）、`cdylib`（给 Android/iOS）、`linkhub-cli`（命令行） |
| 加密 | `ed25519-dalek`（身份签名）、`x25519-dalek`（密钥交换）、`snow`（Noise KK 握手 + ChaCha20-Poly1305 AEAD）、`sha2`（文件完整性） | 身份用 Ed25519，传输用独立的 X25519 静态密钥对 |
| 发现 | `mdns-sd` | 服务名 `_linkhub._tcp.local.`，TXT 含协议版本/设备ID/名/指纹/端口 |
| 安全存储 | Windows: **DPAPI**（`windows-sys`，已实现）；macOS: `security-framework`（依赖已声明，**未落地**）；Linux: `secret-service`（依赖已声明，**未落地**） | `secure:<path>` 前缀触发安全身份文件 |
| 桌面端 | **Tauri 2** + 原生 HTML/JS 前端 + `qrcode`（SVG 二维码） | 前端无框架，靠 `window.__TAURI_INTERNALS__.invoke` 调命令 |
| Android | **Kotlin + Jetpack Compose**（Material3）、CameraX + ML Kit/ZXing（扫码）、`security-crypto`（加密 SharedPreferences）、Gson、Coroutines；通过 **JNI** 调 `liblinkhub_core.so` | minSdk 26 / targetSdk 34，前台服务 dataSync |
| iOS | SwiftUI + Bonjour + Rust FFI（仅源码草稿） | 无 `.xcodeproj`，不能构建 |
| 构建/环境 | Cargo、Gradle（AGP + Kotlin）、Node.js（Tauri 前端工具链）、VS Build Tools、JDK 17、Android SDK/Build Tools 34、AVD `LinkHub_API_34`、NDK 28.2 + cargo-ndk（交叉编译 `.so`） | 见 [环境部署.md](环境部署.md) |

## 4. 核心模块关系

```
                    ┌─────────────────────────────┐
                    │      core-rs (Rust)         │
                    │  身份/配对/信任 · 加密 ·     │
                    │  mDNS 发现 · TCP 收发 ·      │
                    │  心跳/路由/在线状态          │
                    └─────────────────────────────┘
                      ▲          ▲            ▲
              lib +   │   cdylib │   cdylib   │   bin
        tauri 调用    │   (.so)  │   (FFI)    │
                      │          │            │
        ┌─────────────┴──┐  ┌────┴──────┐  ┌──┴──────────┐  ┌───────────┐
        │ desktop        │  │ android   │  │ ios (草稿)  │  │ linkhub-  │
        │ (Tauri 外壳)   │  │ (JNI)     │  │ (FFI 草稿)  │  │ cli       │
        │ HTML/JS 前端   │  │ Compose   │  │ SwiftUI     │  │ 命令行     │
        └────────────────┘  └───────────┘  └─────────────┘  └───────────┘
```

- **唯一事实来源是 `core-rs`**：身份生成、配对确认码、Noise 握手、文件分片/续传、mDNS 全部只在 Rust 里实现一次。
- **桌面端**：`desktop/src-tauri/src/main.rs` 把 core 包成 20 个 `#[tauri::command]`（如 `pairing_generate_qr` / `send_encrypted_text` / `start_listener` / `scan_trusted_mdns`），前端 JS 通过 IPC 调用。桌面端用 `path` 依赖直接链接 core 源码。
- **Android**：`RustBridge.kt` 声明 `external` 函数，对应 `core-rs/src/jni_bridge.rs`；core 编译出的 `.so` 提交在 `android/app/src/main/jniLibs/`（arm64-v8a + x86_64），现已可用 NDK 28.2 + cargo-ndk 复现重建。Kotlin 端只做 UI、前台服务、加密存储、扫码；收到文件时由 native 回调 `RustBridge.onFileReceived` → 通知 + 接收历史。
- **iOS**：`RustBridge.swift` + `ios_bridge.rs` 是镜像设计，但没有 Xcode 工程把它们粘起来。
- **CLI** 是功能最全、最早成型的入口，也是各平台行为的「合同」与验收基准。

**关键设计约定**：发现到一个设备 ≠ 已连接；只有设备已在 Trust Store 中，发现结果才会被转成 `LAN_TCP` 候选路由。配对靠双方各自算出的「确认短码」一致来建立信任。

## 6. 潜在问题和风险

**高优先级**
1. **Android↔Windows 文件互发尚待真机验收**（代码四层已就绪，Android 收到文件也已有通知+历史反馈；剩两台真机/模拟器实跑，见 [真机测试指南.md](真机测试指南.md)）。
2. **`.so` 仍被 git 跟踪**：现已可用 NDK 28.2 + cargo-ndk 复现重建（已消除「无法重建」风险），但二进制仍在仓库里，理想是改为构建产出而非提交。
3. **macOS / Linux 安全存储未落地**：依赖已在 Cargo.toml 声明，但 `identity.rs` 注释明确说「仍需补齐 Keychain / Secret Service 后端」。非 Windows 桌面端目前没有等价的安全身份保护。

**中优先级**
4. **桌面端大量硬编码路径**：`app.js` 默认 `C:\LinkHub\...`、`main.rs` 里 `HISTORY_PATH` 写死。非 Windows 平台直接不可用，违背「跨平台」定位。
5. **桌面前端是无框架原生 JS**：随功能增长会难维护，状态散在 localStorage（含 peer 地址）。
6. **iOS 名不副实**：目录存在但不可构建，容易误导接手者以为 iOS 已起步。
7. **巨型文件**：`identity.rs`(1499) / `net.rs`(1578) / `main.rs`(1593) / `protocol.rs`(794) 单文件偏大，关注点耦合，重构成本随时间上升。
8. **mDNS 多网卡 / 真机网络刷新**：路线图标注「地址刷新需更多真机验证」「多网卡策略待优化」，发现到的地址自动回填 Trust Store 还没做。

**低优先级 / 卫生**
9. Android 存储迁移缺 instrumentation / Robolectric 测试。
10. 测试集中在 core 单测和 Tauri smoke test；跨进程/跨设备无自动化（靠手动脚本）。

3. **拆分巨型文件**（`identity.rs` / `net.rs` / `main.rs`），按身份、配对、信任存储、会话、协议分模块。
