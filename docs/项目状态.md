# LinkHub 项目状态

更新时间：2026-06-16

## 已确认可用

- Rust core 测试通过。
- Tauri desktop Rust 测试通过。
- Android Debug APK 构建通过。
- Android API 34 模拟器 `LinkHub_API_34` 可启动、安装 APK 并打开应用。
- Windows 本机已配置 Rust、JDK 17、Node.js、Visual Studio Build Tools、Android SDK/AVD。

## 本轮新增（2026-06-16 · core 拆分与跨进程测试）

- core 认证监听入口拆分：`net.rs` 中的认证 listener 公共入口、文件接收事件和回调类型迁移到 `net/auth_listener.rs`；`net.rs` 继续 re-export 原有符号，`lib.rs` 公共 API 导出保持不变。
- 新增 cargo 友好的跨进程集成测试：`core-rs/tests/authenticated_cli_e2e.rs` 会启动真实 `linkhub-cli` 子进程，覆盖身份初始化、payload 配对、双向信任写入、`listen-auth`、`send-text-auth`、`send-file-auth`，并校验接收文件内容。
- 验收：`core-rs cargo fmt --all --check`、`core-rs cargo check`、`core-rs cargo test --quiet`、`desktop/src-tauri cargo check`、`core-rs cargo ndk -t arm64-v8a -t x86_64 check --lib`、`scripts/verify-local-e2e.ps1` 均通过；当前仅保留既有 `net/protocol.rs` 未使用 helper 警告。

## 本轮新增（2026-06-16 · 桌面端跨平台路径）

- 桌面端去硬编码：Tauri 新增 `default_config` 命令，按操作系统返回应用数据目录下的默认路径（Win `%APPDATA%\com.linkhub.desktop`、macOS `~/Library/Application Support/...`、Linux `~/.local/share/...`），不再写死 `C:\LinkHub`；`history.json` 的兜底路径也改为跨平台计算。
- 前端启动时拉取 `default_config`，仅在设置尚未保存时 seed（已存设置不动，Windows 老环境零影响），并重建配对页设置输入。
- 身份默认：Windows 仍用 `secure:`（DPAPI）；非 Windows 暂用明文身份文件（待 macOS Keychain / Linux Secret Service 后端落地后再切安全存储）。
- 验收：桌面 `cargo check` 通过、`app.js` 语法检查通过（本机为 Windows，macOS/Linux 实跑待相应环境验证）。

## 本轮新增（2026-06-16 · 地址自动发现）

- 可信设备地址自动发现/自动回填：把原来手动一次性的 mDNS 扫描升级为后台周期刷新。
  - 桌面端：前端后台轮询（~8s）调 `scan_trusted_mdns` 自动回填地址，设备页显示「在线 · IP:port · N 秒前 / 离线」，发送页选中设备地址自动更新（保留手动编辑）。
  - Android：设备页/发送页屏幕级周期自动扫描（~10s）调 `scanTrustedMdnsPeers` + `updatePeerAddress`，设备页显示在线状态，发送页地址自动回填，不再必须手点「扫描并填入地址」。
  - 未改 core/mDNS 协议，全部复用现有扫描与回填件；手动「Scan LAN / 扫描局域网」按钮保留。
- 验收：桌面 JS 语法检查通过、Android `assembleDebug` 通过、模拟器运行时冒烟无崩溃；mDNS 跨设备地址自动出现需真实局域网两台设备验收（模拟器 NAT 下多播不通）。

## 本轮新增（2026-06-16 · 文件接收反馈）

- Android 收到文件后的用户反馈：core 认证监听器新增 `run_authenticated_listener_on_with_callback` + `ReceivedFileEvent`，文件接收并 SHA-256 校验后回调；JNI 在监听线程回调 Kotlin `RustBridge.onFileReceived`，前台服务接到后弹「已收到文件」通知 + 写 `direction=received` 历史；历史卡片新增 ↑发送/↓接收 标识。
- 可复现的 Android `.so` 构建：补齐 NDK 28.2 + Rust 安卓 target + cargo-ndk，`core-rs` 可交叉编译 arm64-v8a / x86_64 并直接产出到 jniLibs；旧接口保持不变，桌面端/Tauri 零影响。
- debug APK 编译通过（gradle assembleDebug，含新 `.so`）。
- 模拟器端到端验收通过（`LinkHub_API_34` + adb 端口转发，Windows CLI 发送方）：Windows→Android 认证加密文件传输完成，Android 收到文件 SHA-256 一致，且**接收回调触发**——历史出现 `direction=received` 记录、通知发出（new JNI 路径 `get_java_vm`/`find_class`/`new_global_ref`/`onFileReceived` 全部验证）。

## 已完成修复

- 修复 JNI listener 和 desktop listener 预绑定后又让 core 重复绑定同一地址的问题。
- core 新增 `run_authenticated_listener_on`，允许上层传入已经绑定好的 `TcpListener`。
- Android 关闭系统备份，身份和信任设备数据迁移到加密 SharedPreferences。
- Android 前台服务会轮询 native listener 状态；后台线程退出或失败时会更新服务页状态并停止 mDNS 广播。
- Android 服务页中文文案已修正，监听状态、接收目录、本机地址和错误信息可直接用于模拟器/真机测试。
- Android 设备页和发送页中文文案已修正，可信设备扫描、地址刷新、文本发送和文件发送状态可读。
- Android NSD 发现已按 Android 14+ 接入 `registerServiceInfoCallback` 和 `hostAddresses`，旧系统保留兼容解析路径。
- 清理误提交的 Android/Gradle/Tauri 构建产物跟踪，并补充 `.gitignore`。
- README 和项目交接入口改为当前文档，不再依赖旧版专用提示词。
