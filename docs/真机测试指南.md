## 2. 当前推荐测试拓扑

先使用最简单可靠的真实环境：

```text
Android 手机开启热点
Windows 连接手机热点
USB 数据线连接手机和 Windows，仅用于 adb 控制、安装、抓日志
LinkHub 数据传输走 Wi-Fi 热点局域网，不走 USB 隧道
```

注意：

- 有线网和 VPN 会影响默认路由判断，真实无线测试时建议先断开有线网。
- 只要 `Test-NetConnection` 显示 `InterfaceAlias: WLAN`，就说明测试流量走的是热点 Wi-Fi。
- ADB 可以保留，因为它只用于调试，不代表 LinkHub 数据链路走 USB。

Windows：

Android：

## 7. 下一步验收

下一轮建议按这个顺序继续：

1. Android -> Windows 文件发送。
2. Windows -> Android 文件发送。
3. 可信设备地址自动更新为最新 mDNS / Wi-Fi 可达地址。
4. 用桌面端 UI 复测 Windows 侧监听、发送、文件选择、接收目录选择和历史记录。

桌面端当前已支持：

- Service 页启动/停止 listener。
- Service 页会在端口被占用或地址非法时显示启动错误。
- Service 页启动 listener 后会自动 mDNS 广播当前桌面端。
- Service 页显示当前网络地址提示。
- Service 页通过 `Browse...` 选择接收目录。
- Send File 通过 `Browse...` 选择本机文件。
- Devices 页通过 `Scan LAN` 手动扫描 mDNS 中已配对可信设备的地址。
- 发送成功后按可信设备记住上次可达地址。
- 发送成功或失败都会写入 Settings/History 页配置的 `historyPath`。

Android 当前也已支持：

- 服务页启动监听后自动通过 Android NSD 广播 `_linkhub._tcp`。
- 服务页会显示真实 listener 状态、接收目录、mDNS 服务名和最近错误。
- 服务页会在端口被占用或监听地址非法时显示启动错误。
- 服务页会显示本机 IPv4 地址提示，可直接给 Windows 或其他设备填写。
- 设备页通过 `扫描局域网` 手动扫描已配对可信设备地址。
- 发送页选择可信设备后，可点 `扫描并填入地址` 直接更新当前目标地址。
- 历史页会记录文本/文件发送成功和失败，便于真机测试后回看。
- 扫描结果只会写回已配对设备，不会把陌生设备加入可信列表。

## 10. mDNS/NSD 发现验证

> 自 2026-06-16 起，桌面端（设备页/发送页）和 Android（设备页/发送页）会在前台**后台周期自动发现**可信设备地址（桌面 ~8s、Android ~10s）并自动回填，设备页显示「在线 · IP:port · N 秒前 / 离线」。下面的手动「Scan LAN / 扫描局域网」按钮仍保留作兜底。真机验收要点：两台已配对设备同网段、各自启动监听后，**不手动点扫描**，对端地址应在一个周期内自动出现且状态转为在线；离开网络后约一个超时周期（桌面 24s/Android 24s）内转为离线。注意模拟器 NAT 下 mDNS 多播不通，自动发现需真实局域网两台设备验收。

Windows 桌面端发现 Android：

1. Android App 打开 `服务` 页。
2. 监听地址保持 `0.0.0.0:8787`。
3. 点击 `启动监听`。
4. 确认服务页显示 `状态: 运行中`，并显示接收目录和 mDNS 服务名。
5. 查看 `本机地址提示`，记录 Android 当前 `IP:8787`，自动发现失败时可手动填这个地址。
6. Windows 桌面端打开 `Devices` 页。
7. 点击 `Scan LAN`。
8. 期望 `Android Phone` 的 Address 更新为 Android 当前 `IP:8787`。

Android 发现 Windows 桌面端：

1. Windows 桌面端打开 `Service` 页。
2. 监听地址使用 `0.0.0.0:8787`。
3. 点击 `Start Listening`，页面应显示 mDNS advertise 状态。
4. Android App 打开 `设备` 页。
5. 点击 `扫描局域网`。
6. 期望 `Windows PC` 的地址更新为 Windows 当前 `IP:8787`。

也可以在 Android `发送` 页选择 `Windows PC` 后，直接点击 `扫描并填入地址`，期望地址输入框自动填入 Windows 当前 `IP:8787`。

## 11. Android listener 状态监控

Android 前台服务启动 listener 后会每秒读取 native `listenerStatus()`：

- native listener 仍在运行时，服务页保持显示运行中，并刷新状态时间。
- native listener 后台线程退出或失败时，服务页会显示 stopped/failed 状态和最近错误。
- listener 失败后会停止 Android NSD mDNS 广播，避免其他设备继续发现一个不可连接的端点。

这个检查已覆盖模拟器安装启动烟测；真机联调时仍建议在端口占用、错误地址和正常启动三种场景各测一次服务页状态。

## 12. Android NSD API 兼容

Android 14+ 设备使用 `registerServiceInfoCallback` 解析服务详情，并通过 `hostAddresses` 获取地址；旧系统继续走 `resolveService` 兼容路径。构建时不应再出现 NSD `resolveService` 或 `host` 的 deprecated 警告。

设备页和发送页已清理乱码文案，真机测试时优先使用：

- 设备页 `扫描局域网` 更新已配对设备地址。
- 发送页 `扫描并填入地址` 更新当前目标地址。
- 发送页文本/文件状态和通知文案确认成功或失败原因。

注意：

- 发现只对已配对可信设备写回地址。
- 如果扫不到，先确认手机热点/路由器允许 mDNS，多播未被 VPN、防火墙或热点隔离阻断。

# 附：文件互发验收（接收反馈版）

> 配套改动：Android 接收文件后新增「通知 + 接收历史」反馈（方案 B：JNI 回调 + 重编 `.so`）。
> 本节在上面真机联调基础上，补充「收到文件给用户反馈」的验收点，以及一种用模拟器 + adb 端口转发的快速验收法。

## A. 本轮代码改了什么

- **core（`net.rs`）**：认证监听器新增 `run_authenticated_listener_on_with_callback`，文件接收并通过 SHA-256 校验后触发 `ReceivedFileEvent` 回调。旧接口保持不变，桌面端不受影响。
- **JNI（`jni_bridge.rs`）**：`startListener` 在 JNI 线程捕获 `JavaVM` 和 `RustBridge` 类全局引用，监听线程收到文件时回调 Kotlin 静态方法 `onFileReceived`（已规避 Android「子线程 FindClass」限制）。
- **Kotlin**：`RustBridge.onFileReceived` → 前台服务注册的处理器 → 写一条 `direction=received` 历史 + 弹「已收到文件」通知；历史卡片新增 ↑发送/↓接收 标识。
- **`.so`**：用 NDK 28.2 + cargo-ndk 重新交叉编译 arm64-v8a 与 x86_64，已替换 `android/app/src/main/jniLibs/`。

## B. 真机验收：Windows → Android（重点）

在第 9 节基础上，**额外确认接收反馈**：

1. 安装本轮新 APK（旧 `.so` 没有回调）：
   ```powershell
   cd C:\Dev\VSCode\LinkHub\android
   adb install -r app\build\outputs\apk\debug\app-debug.apk
   ```
2. 首次安装后打开 App 并授予通知权限（否则只记历史、不弹通知）。
3. 两端互信后（接收方 Android 必须信任发送方 Windows），Android 服务页启动监听，Windows 用 `send-file-auth` 发文件。
4. **预期结果（本轮新增）**：
   - Android 弹通知「已收到文件: <设备名>: <文件名> (大小)」。
   - Android「历史」页出现 `↓ 接收 文件 · <设备名>`，状态成功，detail 显示大小和落盘路径。
   - 文件落在 `Android/data/com.linkhub.app/files/Download/LinkHub/`。
5. 完整性校验：
   ```powershell
   adb shell sha256sum /sdcard/Android/data/com.linkhub.app/files/Download/LinkHub/<file>
   ```
   与源文件 `Get-FileHash <file> -Algorithm SHA256` 比对一致。

## C. 模拟器快速验收法（无需第二台真机）

> 已于 2026-06-16 在 `LinkHub_API_34`（android-34 x86_64）上按本法跑通：Windows→Android 认证加密文件传输完成，Android 端 SHA-256 与源文件一致，历史出现 `direction=received`、通知发出。
> 2026-06-16 回归补充：`.so` 改为构建产物后，`scripts/build-android-so.ps1`、`assembleDebug`、无头模拟器安装启动均已复验通过；但在 headless adb 自动化里，手动配对输入框对长 `linkhub-pair-v1|...` payload 不稳定，认证传输自动复验建议等待 Android 扫码导入配对码功能落地后再跑，或在 Android Studio 可视化模拟器/真机上手动完成配对。
> 2026-06-16 扫码补充：Android 配对页已支持 CameraX + ML Kit 扫码导入对方 payload；无头 AVD 已验证相机权限和扫码预览入口可打开，真实 QR 识别请用真机或带摄像头画面源的 AVD 复验。

没有第二台真机时，可用模拟器当 Android 侧、Windows CLI 当发送侧，靠 adb 端口转发打通：

1. 无头启动模拟器并装 APK：
   ```powershell
   $sdk='C:\Users\<用户名>\AppData\Local\Android\Sdk'
   Start-Process "$sdk\emulator\emulator.exe" -ArgumentList @('-avd','LinkHub_API_34','-no-window','-no-snapshot','-gpu','swiftshader_indirect','-no-audio')
   adb wait-for-device
   adb install -r C:\Dev\VSCode\LinkHub\android\app\build\outputs\apk\debug\app-debug.apk
   adb shell pm grant com.linkhub.app android.permission.POST_NOTIFICATIONS
   ```
2. 模拟器内 Android 服务页启动监听 `0.0.0.0:8787`，并完成与 Windows 的双向配对。
3. 端口转发，让 Windows 用 `127.0.0.1:8787` 直达模拟器：
   ```powershell
   adb forward tcp:8787 tcp:8787
   ```
4. Windows CLI 发文件到 `127.0.0.1:8787`（见 [../scripts/verify-local-e2e.ps1](../scripts/verify-local-e2e.ps1) 的发送段落）。
5. 断言：
   ```powershell
   adb shell ls /sdcard/Android/data/com.linkhub.app/files/Download/LinkHub
   adb shell run-as com.linkhub.app cat shared_prefs/linkhub_history.xml   # 应含 direction=received
   adb logcat -d | Select-String "onFileReceived|Authenticated file end"
   ```

> 说明：mDNS 在模拟器 NAT 下不通，所以这里用 adb 端口转发直连；真机走同网段热点时按上面真机章节用真实 IP。

## D. 通过标准

- [ ] Windows → Android 文件成功落盘，**Android 弹通知 + 历史出现 ↓接收 记录**。
- [ ] Android → Windows 文件成功落盘，Android 历史出现 ↑发送 记录。
- [ ] 两个方向 SHA-256 均一致。
- [ ] （可选）断点续传后文件完整（`Resuming partial file` 日志）。

## E. 排查

| 现象 | 可能原因 |
|------|----------|
| 发送返回 `AUTH_UNTRUSTED` | 接收方可信库没有发送方，补该方向配对 |
| 连接超时 | 不在同一网段 / 防火墙挡了 8787 / 地址填错；真机优先用热点，模拟器用 `adb forward` |
| 文件到了但没通知 | Android 未授予通知权限（历史仍会记录，去「历史」页确认） |
| 收不到文件回调 | 确认装的是本轮新 APK；`adb logcat | grep onFileReceived` 看是否有 JNI 报错 |

## F. 已知限制 / 后续

- 接收目录是 App 专属外部目录（`Android/data/.../files/Download/LinkHub`），普通文件管理器可访问，但不是公共下载目录。若要落到公共 Downloads，需要 MediaStore/SAF 改造（后续任务）。
- `onFileReceived` 仅在 debug 构建验证；若日后开启 R8/混淆，需加 keep 规则保留该方法（被 native 反射调用）。
