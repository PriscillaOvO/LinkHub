# LinkHub 技术架构

## 1. 系统形态

每个客户端都内置一个轻量级 Device Agent。该代理负责设备身份、设备发现、心跳、传输选择和同步队列。

```text
Device Agent
  Discovery Manager
  Presence Manager
  Transport Manager
  Mesh Router
  Trust Manager
  Sync Engine
  Policy Engine
```

## 2. 核心运行时选择

共享核心使用 Rust 实现。

选择 Rust 的原因：

- 跨平台支持较好
- 网络和加密生态成熟
- 并发模型安全
- 可以嵌入 Android、iOS、Windows 和 macOS 客户端

平台外壳建议：

```text
Android: Kotlin + Rust FFI
iOS/iPadOS: Swift + Rust FFI
Windows: Tauri 或原生外壳 + Rust Core
macOS: Tauri 或 Swift 外壳 + Rust Core
```

## 3. 发现层

设备发现应组合多种来源：

- 使用 mDNS / Bonjour 进行局域网发现
- 使用 BLE 广播提供附近感知和低功耗提示
- 使用云端在线状态服务提供跨网络感知
- 使用二维码或短码完成可信关系初始化

发现事件不应直接等同于“已连接”。它只表示某个设备出现了新的可达候选路径。

当前 Rust 原型已经加入发现数据层和基础 mDNS 运行时：`discovery.rs` 定义 `_linkhub._tcp.local.` 服务名、TXT 字段、发现端点、发现时间和 TTL 注册表，`mdns_runtime.rs` 使用 `mdns-sd` 进行真实广播和浏览。TXT 字段包含 `lh` 协议版本、设备 ID、设备名、身份指纹和 TCP 端口。发现结果只有在设备已经处于可信设备列表中时，才会被 `DeviceAgent` 转换成 `LAN_TCP` 候选路由；未配对设备会保留给后续 Trust Manager 和配对流程处理。

## 4. 在线状态与心跳

在线状态按设备和传输通道分别跟踪。

建议状态：

```text
Unknown
Discovered
Pairing
Trusted
Connecting
Connected
Degraded
Reconnecting
Offline
```

心跳策略：

- 前台局域网心跳：2 秒
- 桌面端后台心跳：5 到 10 秒
- 移动端后台心跳：受平台限制，采用机会性策略
- 云端在线状态心跳：30 到 60 秒

## 5. 传输优先级

默认路由优先级：

```text
LAN QUIC
LAN TCP
WebRTC P2P
Bluetooth LE control channel
Cloud Relay
Offline Queue
```

传输管理器应基于评分选择路由，而不是完全依赖固定顺序。评分因素示例：

- 延迟
- 带宽估计
- 电量成本
- 计费网络成本
- 可靠性
- 平台可用性
- 用户策略

## 6. Mesh 模型

MVP 阶段优先实现直接点对点路由，生产级 Mesh 后续再加入。

未来每个可信设备可以广播：

- 已知节点
- 路由质量
- 最近心跳时间戳
- 是否允许中继
- 电量和网络策略

多跳转发必须经过授权，并保持端到端加密，使中继设备无法读取 payload 内容。

## 7. 安全模型

可信关系从二维码或短码配对开始。

每台设备应拥有：

- 设备 ID
- 长期身份密钥
- 每个会话的传输密钥
- 可信设备图谱成员关系

推荐基础加密原语：

- 使用 TLS 1.3 或 Noise Protocol 建立安全通道
- 使用 Ed25519 表示设备身份
- 使用 X25519 进行密钥交换
- 使用 ChaCha20-Poly1305 或 AES-GCM 加密 payload

所有敏感同步 payload 都应端到端加密。

当前 Rust 原型已经加入配对信任骨架：`identity.rs` 定义设备身份、Ed25519 本机身份密钥骨架、公钥指纹、配对邀请、可二维码化的 `linkhub-pair-v1` 邀请 payload、双方确认短码、可信设备记录、可保存/读取的 Trust Store 文件、可生成/读取的本机身份文件，以及对 `linkhub-auth-v1` 握手挑战的签名和验证。Windows 上已经支持 `linkhub_secure_local_identity_v1` 安全身份文件，使用当前用户 DPAPI 保护序列化后的本机身份；CLI 通过 `secure:<path>` 读取安全身份文件，认证 listener/sender 和配对命令都可以复用。CLI 层已经提供 `pairing-code` 和 `trust-pairing`，可以用本机身份 + 对端 payload 计算确认码，并在确认码匹配后写入 Trust Store，作为后续二维码/扫码 UI 的 core 合同。TCP 层已经有认证版文本和文件发送原型，监听端会先检查 Trust Store，再用对端 public key 验证签名，通过后才接受文本和文件分片。确认短码由双方身份指纹和一次性 nonce 派生，双方看到同一个短码才会把对端加入可信设备列表。当前认证通道已通过 Noise KK 握手（`snow` crate）实现双向认证和 ChaChaPoly AEAD 加密传输：发送端和监听端各生成 X25519 静态密钥对（与 Ed25519 签名密钥并存），配对时通过 `linkhub-pair-v1` payload 交换 X25519 公钥；AUTH_OK 后发起 Noise KK 握手，双方用 TrustStore 中的 X25519 公钥完成密钥协商，后续所有文本和文件传输均以 2 字节长度前缀的加密帧进行。身份文件已加入 `dh_key` 字段，TrustStore 格式已更新为 5 字段。当前仍需补齐 macOS Keychain / Linux Secret Service 后端和二维码渲染 UI。

## 8. 初始组件

```text
core-rs/
  src/
    lib.rs
    device.rs
    discovery.rs
    identity.rs
    mdns_runtime.rs
    presence.rs
    routing.rs
    transport.rs
    net.rs
    net/
      ack.rs
      auth_listener.rs
      file_transfer.rs
      protocol.rs
    main.rs
```

当前启动原型已经覆盖在线状态、传输评分、路由选择、连接状态模拟、Ed25519 本机身份生成和持久化、Windows DPAPI 安全身份文件、配对邀请 payload 生成和解析、基于 payload 的确认码计算和信任写入、握手挑战签名和验证、认证版 TCP 文本和文件发送、配对短码确认、可信设备 Trust Store 文件保存/读取、mDNS TXT 载荷生成和解析、基础 mDNS 广播和浏览、基于 Trust Store 的可信 mDNS 扫描、发现结果 TTL 注册、可信设备发现过滤、最小 TCP 心跳链路、带 `message_id` 匹配 ACK、超时重试和重复消息去重的 TCP 文本消息，以及基于 `FILE_START` / `FILE_CHUNK` / `FILE_END` 的小文件传输、稳定文件 `transfer_id`、可配置接收目录、`.part` 临时文件落盘、可写入和解析的 `.part.meta` 接收进度元数据、SHA-256 完整性校验，以及通过 `FILE_START_RECEIVED:<chunk_index>` 协商起点的基础断点续传。续传复用半成品前会校验 `.part` 实际长度、元数据字节数、完整 chunk 边界和半成品 SHA-256。Noise KK 双向认证和 ChaChaPoly AEAD 加密传输位于 `crypto.rs`（使用 `snow` crate），基于 X25519 静态密钥对的 Noise 握手在 AUTH_OK 后自动执行，后续所有 TCP 帧均以加密帧格式（2 字节长度前缀）传输。配对信任骨架位于 `identity.rs`，发现数据层位于 `discovery.rs`，mDNS 运行时位于 `mdns_runtime.rs`，TCP 会话流程仍在 `net.rs`，认证 listener 公共入口和文件接收回调类型位于 `net/auth_listener.rs` 并由 `net.rs` re-export 旧 API，ACK 重试辅助逻辑位于 `net/ack.rs`，线协议编解码已经拆到 `net/protocol.rs`，文件传输 ID、路径、分片大小、接收进度元数据和 SHA-256 辅助逻辑已经拆到 `net/file_transfer.rs`。跨进程 CLI 认证发送集成测试位于 `core-rs/tests/authenticated_cli_e2e.rs`，后续会继续加入 macOS/Linux 安全存储、二维码渲染 UI 和 ffi 等模块。
