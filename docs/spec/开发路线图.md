# LinkHub 开发路线图

## 阶段 0：原型基础

目标：验证本地 Device Agent 模型。

- 实现设备身份模型
- 实现心跳时间戳跟踪
- 实现连接状态转换
- 实现传输通道评分
- 模拟断连、降级和恢复

交付物：命令行 Rust 原型。

## 阶段 1：局域网 MVP

目标：让两台桌面设备可以在同一网络中互相发现并传输数据，为长期可信设备 Mesh 打地基。

- mDNS 发现
  - 已完成：发现端点模型、mDNS TXT 载荷合同、基础 mDNS 广播和浏览、Trust Store 可信扫描接入、TTL 注册表、可信设备发现过滤
  - 待完成：跨设备联调、平台权限提示、多网卡策略优化
- TCP 或 QUIC 控制通道
- 加密配对
  - 已完成：设备身份、Ed25519 本机身份密钥骨架、公钥指纹、确认短码、Trust Store 文件保存/读取、本机身份文件保存/读取、Windows DPAPI 安全身份文件、可二维码化的配对邀请 payload、基于 payload 的确认码和信任写入命令、握手挑战签名和验证骨架、认证版 TCP 文本和文件发送原型
  - 待完成：二维码渲染 UI、macOS Keychain / Linux Secret Service 安全存储后端、双向认证、会话密钥协商、加密传输
  - 待加固（xhigh code-review 2026-06-17 暴露，多为既存）：① 配对 payload 加**绝对签发时间戳**，让 TTL 在接收端真正可校验（现在 `from_payload`/`confirm` 都用 `Instant::now()`，`is_expired` 永远 false）；② **加长确认码**（现仅 24 位/6 hex，去 nonce 后可被中间人离线撞码——不可用"加回 nonce"修，否则重新弄坏两端码一致）；①②合并为 `linkhub-pair-v2` 格式（顺手删已无安全作用的 nonce 字段）。建议动手前对配对安全设计单跑 `/code-review ultra`。
- 两个 agent 之间的心跳
- 文本发送
- 文件发送和基础断点续传
- 连接状态 UI

推荐第一个客户端选择 Windows 桌面端，因为它最容易运行、检查并保持常驻。
