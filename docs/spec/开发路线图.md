# LinkHub 开发路线图

## 阶段 0：原型基础

目标：验证本地 Device Agent 模型。

- 实现设备身份模型
- 实现心跳时间戳跟踪
- 实现连接状态转换
- 实现传输通道评分
- 模拟断连、降级和恢复

交付物：命令行 Rust 原型。

## 阶段 1：局域网 MVP

目标：让两台桌面设备可以在同一网络中互相发现并传输数据，为长期可信设备 Mesh 打地基。

- mDNS 发现
  - 已完成：发现端点模型、mDNS TXT 载荷合同、基础 mDNS 广播和浏览、Trust Store 可信扫描接入、TTL 注册表、可信设备发现过滤
  - 待完成：跨设备联调、平台权限提示、多网卡策略优化
- TCP 或 QUIC 控制通道
- 加密配对
  - 已完成：设备身份、Ed25519 本机身份密钥骨架、公钥指纹、确认短码、Trust Store 文件保存/读取、本机身份文件保存/读取、Windows DPAPI 安全身份文件、可二维码化的配对邀请 payload、基于 payload 的确认码和信任写入命令、握手挑战签名和验证骨架、认证版 TCP 文本和文件发送原型
  - 待完成：二维码渲染 UI、macOS Keychain / Linux Secret Service 安全存储后端、双向认证、会话密钥协商、加密传输
  - 待加固（xhigh code-review 2026-06-17 暴露，多为既存）：① 配对 payload 加**绝对签发时间戳**，让 TTL 在接收端真正可校验（现在 `from_payload`/`confirm` 都用 `Instant::now()`，`is_expired` 永远 false）；② **加长确认码**（现仅 24 位/6 hex，去 nonce 后可被中间人离线撞码——不可用"加回 nonce"修，否则重新弄坏两端码一致）；①②合并为 `linkhub-pair-v2` 格式（顺手删已无安全作用的 nonce 字段）。建议动手前对配对安全设计单跑 `/code-review ultra`。
- 两个 agent 之间的心跳
- 文本发送
- 文件发送和基础断点续传
- 连接状态 UI

推荐第一个客户端选择 Windows 桌面端，因为它最容易运行、检查并保持常驻。

## 阶段 2：桌面端闭环

目标：让 Windows、Linux 和 macOS 先形成稳定桌面端互联闭环。

- Windows 基础客户端
- Linux 桌面端运行支持
- macOS 桌面端运行支持
- 已完成：桌面端去硬编码，按操作系统使用应用数据目录的默认路径（`default_config` 命令），不再写死 `C:\LinkHub`；macOS/Linux 实跑待相应环境验证
- 已完成：桌面端全量中文化（标签页/表单/按钮/提示）
- 已完成：修复桌面端 UI 卡顿（阻塞型 Tauri 命令从主线程同步 `fn` 改为 `async fn`，后台 mDNS 自动发现不再冻结窗口/按钮）
- 已完成：本机身份持久化 CLI，支持生成/读取稳定设备身份
- 已完成：Windows DPAPI 安全身份持久化 CLI，支持 `secure:<path>` 参与配对和认证命令
- 已完成：本机状态快照和 HTML 状态页，展示本机身份和可信设备列表，作为 Windows 基础客户端首页的数据合同
- 已完成：交互式可信设备列表 UI，支持复制设备信息、按设备扫描刷新、跳转发送页并预选目标
- 传输历史和失败原因
- 已完成：本机双进程自动验收脚本，覆盖文本发送、完整文件发送、预置半成品断点续传、Windows DPAPI 安全身份文件、基于 payload 的配对确认、认证版文本发送、认证版文件发送和 HTML 状态页生成
- 已完成：core 认证 listener 入口拆分到 `net/auth_listener.rs`，`net.rs` 继续 re-export 旧公共 API；新增 cargo 集成测试用真实 `linkhub-cli` 子进程覆盖认证文本和认证文件跨进程发送。
- 已完成：core 巨型文件按关注点拆分（`identity.rs` → `identity/{device_identity,pairing,trust_store,secure_store}`；`net.rs` 会话流程 → `net/{session,auth_session}`），公共 API re-export 保持不变；新增进程内 cargo 集成测试 `tests/e2e.rs`（真实 TCP loopback 覆盖明文文件、认证文本/文件/续传，断言 SHA-256 与回调）。
- 已完成：桌面端 Windows 安装包打包配置（`bundle.targets` = nsis/msi、补 `icon.ico`、官方 schema）+ 单实例守护 + 系统托盘；新增 `scripts/build-desktop-installer.ps1`。已在 Windows 构建出 NSIS `LinkHub_0.1.0_x64-setup.exe` 和 MSI `LinkHub_0.1.0_x64_en-US.msi`，待用户安装后验证单实例和托盘交互。代码签名留接入点（`certificateThumbprint`）未实做。
- 两台真实桌面设备局域网联调

## 阶段 3：Android 集成

目标：加入第一个移动端节点。

- 已完成：Android UI 外壳，包含配对、设备、发送、服务四个页面
- 已完成：Rust core JNI 桥接，支持身份、配对 payload、发送文本/文件、启动/停止 listener
- 已完成：用于稳定测试的前台服务，真机上可启动和停止 `0.0.0.0:8787` 监听
- 已完成：修复监听「停止后无法重启/重绑」竞态——`jni_bridge` 引入 `stop_and_join_listener` 收口（停止时 join 工作线程释放 socket）+ `LISTENER_EPOCH` 代际保护，停止/启动状态始终一致（待真机复验）
- 已完成：修复 release（R8）版生成身份/读列表失败——`proguard-rules.pro` 显式 keep 全部 Gson DTO 类及成员（待真机复验）
- 已完成：Android 本机身份和可信设备持久化
- 已完成：Android 生成 Rust core 可读取的 Trust Store 文件
- 已完成：Windows CLI 与 Android 真机通过手机热点 Wi-Fi 完成双向加密文本发送
- 已完成：Windows -> Android 真实局域网认证发送，返回 `AUTH_OK`
- 已完成：Android -> Windows 真实局域网认证发送，Windows 收到加密文本
- 已完成：Android 配对页生成可扫码二维码，同时保留 payload 文本复制
- 已完成：Android 13+ 通知权限请求、文件选择器、默认 App 专属接收目录、发送结果通知
- 已完成：Android 收到文件后的通知 + 接收历史反馈（core 回调 + JNI `onFileReceived`）
- 已完成：可复现的 Android `.so` 交叉编译（NDK 28.2 + cargo-ndk）；native 二进制已从 git 跟踪移出，按 `scripts/build-android-so.ps1` 生成
- 已完成：Android release 签名打包配置（keystore.properties / 环境变量驱动的 signingConfig + minify/shrink + `proguard-rules.pro` keep JNI 桥）+ `scripts/build-android-release.ps1`；密钥与 `.aab`/`.jks` 均 git 忽略。已用本机 dev keystore 实出**签名 release APK**（`apksigner verify` 通过，V2 签名；APK 含 arm64-v8a/x86_64 `liblinkhub_core.so`），运行期不丢类待真机复验。
- 已完成：**Android↔Android 双向加密文本/文件双模拟器实跑验收**（两个 AVD，adb forward/reverse 打通网络，adb 驱动 Compose UI）——A→B/B→A 文本+文件均成功、接收文件 SHA-256 一致、接收通知 + 收发历史正确。验证过程发现并修复 4 个 bug：①进程被杀后持久化 `service_status.running` 残留致「启动监听」被禁用、监听永不重绑（`ServiceScreen` 改用进程内 `LinkHubService.isRunning`）；②NSD 后台扫描 `RejectedExecutionException` 闪退（executor 改 `DiscardPolicy`）；③release(R8) 版可信设备列表泛型 `TypeToken` 被剥离致不持久化（proguard 加 TypeToken keep，已在 release 验证设备页恢复为 1）；④监听 worker `catch_unwind` 防御假运行态。
- 已修复：`LinkHubService.isRunning` 加 `@Volatile`（唯一存活判据、跨线程读写，缺 volatile 可读到过期值再触发死锁）。
- 待加固（xhigh code-review 2026-06-17）：① core 设一次性 `std::panic::set_hook` + android_logger，让原生 panic 进 logcat（现 `catch_unwind` 只护 accept 线程，每连接 worker 的 panic 仍不可见，是今天原生 bug 难查的根因）；② 停止把 `service_status.running` 持久化作为存活源（现只停了"读"未停"写"，未来读者会重踩死锁）。
- 待完成：Android 与 Windows 文件互发真机实跑验收（已备应用级真机联调清单 `scripts/two-device-app-test.md`，待用户在真机上跑通并回填结果）
- 已完成（UX/安全）：修复配对确认码两端不一致——确认码改为只依赖排序后的双方指纹（去掉只存在于对方 payload 的 nonce），双模拟器验证两端一致。
- 已完成：Android 相机扫码导入对端配对码（CameraX + ML Kit，真机扫码识别待带摄像头画面环境复验）
- 已完成：局域网发现结果自动更新可信设备地址（桌面/Android 后台周期自动发现 + 在线状态 + 地址自动回填；mDNS 跨设备实测待真机两台验收）
- 待完成：公开下载目录等高级接收目录权限处理

当前阶段 3 的准确状态：已经不是单纯骨架，Windows/Android 真实设备文本互发已跑通；但仍不能视为完整移动端产品，下一步应优先验证文件发送和地址自动发现。

## 阶段 4：iOS/iPadOS 集成

目标：加入 Apple 移动端点，并真实处理后台行为限制。

- iOS/iPadOS 应用外壳
- Bonjour 发现
- 本地网络权限处理
- 文档选择器集成
- 分享扩展
- 在产品行为中明确记录后台限制

## 阶段 5：多路径连接

目标：让设备根据网络情况组合不同连接方式，保持更高可达性。

- Wi-Fi 和有线网络优先
- WebRTC 跨网络直连
- 蓝牙附近发现和低速控制
- NFC 快速配对和近场确认
- 云端在线状态服务
- WebRTC 信令
- 中继兜底
- 离线队列同步
- NAT 穿透指标

## 阶段 6：Mesh 路由和授权中继

目标：允许可信设备共享可达信息，并在授权情况下通过其他节点转发。

- Gossip 在线状态交换
- 路由表
- 中继权限模型
- A 到 B、C 到 A、C 经 A 到 B 的多设备转发模型
- 多跳端到端加密转发
- 路由质量评分
- 中继设备不可读取 payload 内容

## 阶段 7：实时音视频

目标：在可信设备之间提供低延迟语音和视频通信。

- 语音电话
- 视频通话
- 通话中设备切换
- WebRTC 或同等级实时媒体栈
- 弱网降级策略

## 阶段 8：媒体播放协同

目标：让音乐和其他媒体可以在可信设备之间无缝迁移和控制。

- 一台设备控制另一台设备播放
- 播放队列同步
- 播放状态同步
- 手机、平板、电脑之间切换播放设备
- 与文件传输和实时通信分离设计
