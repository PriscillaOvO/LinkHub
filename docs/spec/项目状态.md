## 本轮新增：2026-06-20 · C6 滑动窗口吞吐优化暂缓

- **C6 ⏭️ 跳过（按交接允许）**：没有在本批次末尾实现滑动窗口分块。原因是它会同时触碰 `auth_session.rs` 的发送/ACK/恢复语义、`protocol.rs` 的能力协商与帧格式、WebRTC/TURN 文件 e2e 的背压边界；当前 T8 的二进制分帧已把线缆体积砍半，现有「逐块发送、逐块 ACK」仍是正确性基线。
- **风险判断**：滑动窗口不是局部性能 patch，需要明确版本协商（例如 `+winN` 能力）、窗口大小/内存上限、乱序/重传策略、断点续传与旧端回退，再补单元测试和三条 WebRTC 文件 e2e SHA 校验。没有这些设计就贸然改容易破坏现有恢复与兼容语义。
- **后续建议**：把 C6 拆成独立吞吐优化任务：先写协议设计小节和窗口状态机单测，再实现 feature-compatible sender/receiver，最后跑 core 默认 + `--features webrtc` 全矩阵（含 `webrtc_cli_e2e` / DataChannel / TURN 三条文件 e2e）。
- **代码状态**：本次 C6 只有文档决策，无源码改动；webrtc/tokio feature gate、默认 Android `.so`、v2 配对均未触碰。

## 本轮新增：2026-06-19 · T8 二进制文件分帧（版本协商 · 去掉 hex 翻倍）

- **T8 ✅**：把加密路径的文件分块从「hex 文本」换成「二进制」，**线缆体积砍半**，并做**版本协商**保证与旧端兼容。
- **为什么能砍半**：Noise 帧本身就是 `u16 大端长度 + 密文`（[auth_session.rs](../../core-rs/src/net/auth_session.rs) 的 `send_encrypted_frame`），每帧已自带长度前缀。原 `FILE_CHUNK` 把原始字节 hex 编码（2× 体积）只是为了塞进「Tab 分隔的文本行」——对分帧毫无必要。
- **新分帧**：[protocol.rs](../../core-rs/src/net/protocol.rs) 加 `WireMessage::FileChunkBin{transfer_id, chunk_index, data: Vec<u8>}`；`serialize_message_bytes` 产出 ASCII 头 `FILE_CHUNK_BIN\t{id}\t{index}\t` + **原始字节**；`parse_binary_frame` 收到帧先看前缀:命中则切前两个 Tab（结构性）、其余字节**原样**作 data（块内的 Tab/换行/NUL/0xFF 全部保真），否则回退按 UTF-8 文本走 `parse_message`。收发帧改走字节路径（`recv_encrypted_frame` 不再假设 UTF-8）。
- **版本协商（每次传输）**：加密接收端在 `FILE_START` 的 ACK 状态尾部加 `+bin` 能力标记（`file_start_ack_supports_bin`）；发送端 `send_encrypted_file_start_with_retries` 现返回 `(续传起点, 对端是否支持二进制)`，**仅当对端带 `+bin`** 才发 `FileChunkBin`，否则回落 hex `FileChunk`。`parse_file_start_ack_status` 容忍 `+bin` 后缀，续传解析不受影响。**明文 TCP 路径**（按行分隔，不能带裸字节）保持 hex 不变 → 全兼容。
- **背压**：仍是现有「逐块停等 ACK」（发一块等 `FILE_CHUNK_RECEIVED` 再发下一块），二进制化不改变流控语义；`u16` 帧上限 64 KB 对 4 KB 块绰绰有余。
- **验收全绿**：core `fmt --check`；`clippy --all-targets -D warnings`（默认 + `--features webrtc`）；默认 `cargo test`（新增 6 个单测：二进制分帧序列化/解析、含 Tab/NUL 保真、文本回退、畸形拒绝、能力探测，以及**内存双工真跑二进制文件 e2e**——9 KB 多块含坏字节逐字节还原）；`--features webrtc` 全过——`webrtc_cli_e2e` / `webrtc_e2e`(DataChannel) / `webrtc_turn_e2e`(强制 TURN 中继) 三个文件传输 e2e 现**自动走二进制路径**且 SHA-256 一致，等于在真实 WebRTC/TURN 上验证了二进制分帧；`cargo ndk -t arm64-v8a -t x86_64 check --lib` 双 ABI（改动在默认构建，未引入新依赖）。
- **未做**：滑动窗口/多块在途（当前停等 ACK 已是正确背压，吞吐优化留作后续）；明文 TCP 路径仍 hex（裸字节无法过按行协议，且明文路径非主线）。

## 本轮新增（2026-06-17 · 修复 Android 监听服务停止后无法重启 + release 版身份创建失败）
