# LinkHub MVP 产品需求文档

## 1. 产品目标

构建一个面向 Android、iOS、iPadOS、Linux、Windows 和 macOS 的跨平台可信设备 Mesh 互联系统。长期目标是让用户的所有可信设备自动发现、自动连接、自动选择最佳链路，并在授权情况下互相中继，从而支持文本、文件、语音、视频和媒体播放协同。

MVP 阶段的目标是先证明最小闭环：已信任设备能够在局域网内持续可见、可达，并可靠传输文本和文件。

## 2. MVP 定位

LinkHub MVP 是完整 Mesh 愿景的第一块地基：跨平台的“设备在线状态与直接传输层”。

首个版本需要证明：

- 设备可以安全配对
- 已配对设备可以自动发现彼此
- 客户端可以通过心跳维持实时在线状态
- 连接状态可以在无需用户操作的情况下自动降级和恢复
- 文件和文本可以通过当前最优路由发送到目标设备
- 后续可以在这套身份、发现、路由和传输基础上扩展中继、语音视频和媒体协同

## 3. 目标平台

- Android 应用
- iOS 应用
- iPadOS 应用
- Linux 桌面客户端
- Windows 桌面客户端
- macOS 桌面客户端

## 4. 第一版用户故事

### 设备配对

作为用户，我可以扫描二维码或输入短码，将两台设备加入同一个可信设备图谱。

当前 core 已支持这一流程的命令行等价物：一台设备生成可二维码化的 `linkhub-pair-v1` payload，另一台设备用本机身份和 payload 计算确认码，用户确认短码一致后写入 Trust Store。后续桌面 UI 需要把这些命令封装成“显示二维码 / 扫码 / 确认短码”的界面。

### 自动发现

作为用户，当两台可信设备位于同一网络时，它们可以自动出现，而不需要手动输入 IP 地址。

### 实时在线状态

作为用户，我可以看到每台可信设备是在线、附近、可通过局域网访问、可通过云端中继访问、处于降级状态，还是已经离线。

### 无感重连

作为用户，当 Wi-Fi 发生变化时，应用可以自动切换到其他传输通道，并在更快的链路恢复后自动切回。

### 快速发送

作为用户，我可以向另一台可信设备发送文件、图片、文本或 URL。

### 长期多跳互联

作为用户，当设备 C 无法直接连接设备 B，但设备 C 可以连接设备 A、设备 A 可以连接设备 B 时，我希望设备 A 可以在授权情况下作为中继，让 C 与 B 完成通信。

### 长期实时通信

作为用户，我希望未来可以在自己的可信设备之间发起语音电话、视频通话，并在不同设备之间切换通话或播放状态。

## 5. MVP 功能范围

### 包含

- 账号或本地身份初始化
- 可信设备配对
- 使用 mDNS 或 Bonjour 进行局域网发现
- 心跳与在线状态维护
- 传输通道评分与路由选择
- 基于局域网的文件和文本传输
- 发送失败后的离线队列
- 基础桌面端和移动端状态页面

当前 core 已提供 `status` 和 `status-html`，可以从安全身份文件和 Trust Store 生成本机状态快照与 HTML 状态页，展示本机身份和可信设备列表。它还不是最终桌面 UI，但可作为 Windows 客户端首页的数据合同和验收起点。

### 暂缓

- 语音电话
- 视频通话
- 无缝音乐播放
- 跨设备播放控制
- 完整远程控制
- 深度 iOS 通知镜像
- 生产级多跳 Mesh 中继
- 投屏
- 将 iPad 作为无线显示器
- 面向终端用户的插件系统

## 6. 体验原则

- 配对后，设备应尽可能自动出现。
- 链路变化不应打扰用户，除非当前操作失败。
- UI 应清晰展示状态，但系统本身应在用户不持续关注的情况下工作。
- 附近设备之间的“发送到设备”体验应尽量接近 AirDrop 的速度和直接感。
- 中继和多跳必须可解释、可授权、可关闭。
- 中继设备不能读取被转发内容，长期设计必须保持端到端加密。

## 7. 重要平台限制

iOS 和 iPadOS 对后台执行、文件系统访问、通知访问和远程控制有严格限制。第一版应将 Apple 移动平台视为安全协作端点，而不是不受限制的系统代理。

## 8. MVP 成功指标

- 同一局域网内，已配对设备发现时间不超过 3 秒
- 前台心跳能在 10 秒内检测断连
- 降级路由切换在 5 秒内完成
- 常见 Wi-Fi 环境下局域网传输速度达到至少 30 MB/s
- 睡眠唤醒或 Wi-Fi 切换后，无需手动操作即可重连

# 附：长期愿景与阶段拆分

> 本节由原《需求澄清》整合而来，给出超出 MVP 的长期方向；原件存档于 [../archive/需求澄清.md](../archive/需求澄清.md)。

## 一句话定义

LinkHub 是一个跨平台可信设备 Mesh 互联系统：让用户自己的 Android、iOS、iPadOS、Linux、macOS、Windows 设备互相信任、自动发现、自动连接，并在不同网络条件下组合局域网、蓝牙、NFC、有线网络、WebRTC、中继等通道，完成文本、文件、语音、视频和媒体播放协同。

> 最终愿景：用户的所有可信设备组成一个私有互联网络。A 直连 B，C 直连 A；当 C 不能直连 B 时，可在授权和端到端加密保护下经 A 中继，实现 C 到 B 的可靠通信。

第一版不追求「大而全」，只先证明一个核心体验：**两台已配对设备在同一局域网自动发现彼此，并可靠收发文本和文件。**

## 愿景能力六层

1. 设备身份层：稳定身份、可信关系、安全密钥。
2. 发现连接层：局域网、蓝牙、NFC、有线、公网信令发现。
3. 路由传输层：直连/中继/多跳之间按质量自动选择与切换。
4. 数据协作层：文本、URL、文件、图片、离线队列。
5. 实时通信层：语音、视频、低延迟实时媒体。
6. 媒体协同层：跨设备无缝播放、播放控制迁移、播放状态同步。

## 长期阶段拆分

- **阶段 A 可信设备直连**：桌面端优先，局域网发现 + 可信配对 + 文本/文件 + 断点续传 + 状态展示。
- **阶段 B 移动端接入**：Android、iOS/iPadOS，处理后台限制，手机/平板与电脑互传。
- **阶段 C 多路径连接**：Wi-Fi/有线主链路、WebRTC 跨网直连、蓝牙附近发现、NFC 快速配对，路由按延迟/带宽/可靠性/电量/成本实时评分。
- **阶段 D Mesh 与中继**：可信设备授权后作为中继，多跳端到端加密，中继不可读 payload。
- **阶段 E 实时音视频**：语音/视频通话、通话中设备切换。
- **阶段 F 媒体播放协同**：跨设备播放控制、队列与状态同步、播放迁移。

## 设计约束

- 支持越多协议是长期方向，不是第一版要求。
- 蓝牙/NFC 适合发现、配对、唤醒、低速控制，不适合大文件主传输。
- 语音视频复杂度高，应在传输基础稳定后再做。
- Mesh 中继必须以授权和端到端加密为前提。
- iOS/iPadOS 后台能力有限，不能按桌面常驻代理设计。
